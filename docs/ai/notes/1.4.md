## Planned feature boundary

`1.4` currently aims to deliver the following product-level changes:

1. authorization levels
- `admin`
- `group_admin`

2. richer login-user model
- email address
- authorization level
- language preference
- time preference
- optional user icon only if it stays clearly non-blocking

3. self-profile editing
- allow a login user to edit their own profile-level fields

4. stronger login-user management
- extend the current security-area management UI for login users

5. log expansion and separation
- distinguish audit-oriented logs from system logs
- include actor identity in system-side operational logs where feasible

6. log navigation cleanup
- decide how logs should sit relative to Settings, Access, and future authz surfaces

## Non-goals for the first `1.4` slice

- MFA
- passkeys
- OIDC / SAML
- retention policy enforcement
- external log shipping
- append-only or WORM guarantees
- fully polished user-avatar system

## Design constraints

These constraints should not be violated during `1.4` work:

- `config.apply` remains admin-only
- `group_admin` is scope-limited to their own group
- policy evaluation must remain deterministic: `action x role x scope -> decision`
- authorization enforcement must not depend on ad-hoc page logic alone
- deny paths must remain auditable
- audit logging must not be suppressible through normal log-level settings
- avoid collapsing auth/access concerns back into generic Settings structure without a clear reason

## Dependency order

Implement `1.4` in this order unless a narrow bugfix forces a smaller detour.

### Phase A: actor model expansion

Goal:
- extend the login-user model so later authz and profile work have stable fields

Includes:
- email address
- role
- language preference
- time preference
- optional placeholder seam for avatar, but do not block on it

Must happen before:
- profile editing
- admin login-user management expansion
- actor-rich logging

### Phase B: role and policy enforcement

Goal:
- make `admin` and `group_admin` real enforced roles, not only data values

Includes:
- route/action vocabulary review
- policy table implementation aligned with `policy.md`
- scope evaluation aligned with `middleware-design.md`
- admin-only `config.apply`

Must happen before:
- claiming `group_admin` support in UI
- trusting any audit trail to reflect real enforced boundaries

### Phase C: self-profile surface

Goal:
- let a login user manage their own profile fields safely

Includes:
- profile read/update route(s)
- UI for own email/language/time settings
- clear separation between self-editable fields and admin-managed fields

Depends on:
- Phase A

### Phase D: login-user management expansion

Goal:
- turn current login-user administration into a fuller management surface

Includes:
- create/edit login users
- change role
- manage language/time defaults if needed
- optionally manage active/inactive status if introduced

Depends on:
- Phase A
- enough of Phase B to avoid editing fields that are not yet enforced

### Phase E: log split and actor traceability

Goal:
- separate audit intent from system/operational logging and attach actor context consistently

Includes:
- audit vs system log boundary clarification
- actor identity in system logs where possible
- mandatory audit events for allow/deny/sensitive reads

Depends on:
- Phase B for real policy outcomes
- Phase A for stable actor fields

### Phase F: navigation and information architecture cleanup

Goal:
- place profile, login-user management, and logs in a coherent UI structure

Includes:
- decide whether logs belong under Settings, Access, or a separate Logs surface
- avoid undoing the `1.3` responsibility split without a stronger replacement

Depends on:
- enough of Phases C, D, and E to know the real surfaces

## Recommended slice plan

If `1.4` must be delivered incrementally, use these slices.

### Slice 1

- login-user schema expansion
- role field support
- admin-only `config.apply`
- minimal enforced policy checks for high-risk actions

### Slice 2

- self-profile page
- self-update API
- language/time preference plumbing

### Slice 3

- expanded login-user management UI
- role editing flows
- group-scope administration rules

### Slice 4

- audit/system log separation cleanup
- actor identity in operational logs
- navigation cleanup for access/log surfaces

## Open design decisions

These should be resolved deliberately before broad UI work:

1. where should profile live?
- candidate: dedicated Access/Auth area
- avoid hiding it as a random Settings subsection if auth is being separated structurally

2. where should logs live?
- audit logs are access/security-adjacent
- system logs are operational
- mixing both under one generic Settings tab may blur responsibility

3. does `group_admin` need a first-class group association on the actor model?
- if yes, define the source of truth explicitly
- if no, define how scope is resolved without ambiguity

4. is user icon in scope?
- keep optional
- do not let media handling block authz/logging work

## Dependency checklist

Use this checklist before starting or merging `1.4` work.

### Data model

- [ ] login-user schema has email
- [ ] login-user schema has role
- [ ] login-user schema has language preference
- [ ] login-user schema has time preference
- [ ] migration/backfill behavior is defined for existing users

### Authz model

- [ ] `admin` and `group_admin` are implemented as real enforced roles
- [ ] action vocabulary is explicit for protected routes
- [ ] resource scope resolution is defined for group-bound resources
- [ ] `config.apply` is admin-only
- [ ] role-change operations are admin-only
- [ ] deny reasons are mapped consistently

### Middleware and enforcement

- [ ] middleware ordering is clear
- [ ] actor extraction is centralized
- [ ] action resolution is centralized
- [ ] resource resolution includes `group_id` where applicable
- [ ] deny paths stop execution after audit write
- [ ] bypass routes are explicitly listed rather than accidental

### Profile and admin UI

- [ ] self-profile edit fields are separated from admin-only fields
- [ ] login-user management UI reflects enforced backend permissions
- [ ] role editing UI exists only where backend enforcement is ready
- [ ] language/time settings have a clear source of truth

### Logging and audit

- [ ] audit log stream is logically separate from operational logs
- [ ] auth success/failure is audited
- [ ] allow/deny outcomes for protected actions are auditable
- [ ] sensitive read actions are auditable
- [ ] system logs include actor context where feasible
- [ ] correlation ID flows through request handling

### Information architecture

- [ ] Access/Auth surfaces do not regress into generic Settings clutter
- [ ] log placement is intentional rather than historical accident
- [ ] Settings remains focused on system/runtime/logging/UI configuration

## Merge safety checklist

- [ ] backend enforcement lands before or with UI claims
- [ ] tests cover at least one allow and one deny path for new authz rules
- [ ] tests cover actor field serialization/update for new profile data
- [ ] tests cover audit emission for newly protected actions
- [ ] docs are updated where operator-visible behavior changes

## Completion signal for `1.4`

`1.4` should be considered meaningfully complete when:

- `admin` and `group_admin` are real enforced roles
- login users carry the minimum profile/authz fields needed for access control
- users can edit their own profile-level settings
- privileged login-user management is expanded beyond bootstrap-only basics
- audit and system logging have a clearer boundary
- high-risk actions are both controlled and traceable
