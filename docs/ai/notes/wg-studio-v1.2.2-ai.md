# wg-studio v1.2.2 作業内容（AI向け）

## 目的
v1.2.2 では、機能追加を広げる前に、直近で発覚した不具合の修正と、将来の派生開発に耐えるための土台整理を行う。  
今回のスコープは以下の3点に限定する。

1. 直近の不具合修正
2. 汎用パーツの Common 化
3. 命名規則の統一

---

## 1. バグ修正

### 対象
今回の検証・実運用で判明した不具合を修正する。

### 優先度
最優先。  
Common 化や命名整理より前に対応すること。

### 代表的な論点
- WireGuard interface (`wg0`) が存在しない場合の挙動
- interface address のドリフト
- route drift（`ens3` に出てしまう問題）
- Docker API version 固定 (`v1.41`) による runtime 制御失敗
- runtime と config のズレを UI が正常と誤認する問題
- endpoint / interface address の責務分離不足
- apply 後に desired state へ収束しないケース

### 期待成果
- 既知の再現済み不具合が潰れていること
- 少なくとも今回踏んだ障害は再発しないこと
- runtime health の最低限の信頼性が上がること

### 完了条件
- 該当不具合について再現 → 修正 → 再検証が完了している
- 修正内容が後続の Common 化で壊れない構造になっている
- 主要な修正内容が changelog / release note に反映できる状態である

---

## 2. 汎用パーツの Common 化

### 目的
wg-studio 固有ではない機能・UI・責務を切り出し、今後の別システムでも使える土台へ整理する。

### 今回 Common 化したい範囲
#### フロントエンド
- ログイン画面
- 画面全体のレイアウト
- サイドバー / ヘッダ / ページタイトル
- KPIカード
- 共通テーブル
- ステータスバッジ
- 共通モーダル / confirm ダイアログ
- ダッシュボードの共通枠組み

#### バックエンド / ドメイン共通
- 認証の共通土台
- 監査ログのモデル / API / 表示基盤
- 共通レスポンス形式
- 共通エラー表現
- UI が参照する共通 settings / session まわり

### 切り出し方針
- WireGuard 固有の概念は Common に入れない
- 再利用可能な責務だけ切る
- 「共通化のための共通化」はしない
- Common は便利箱にしない
- 将来別製品へ横展開できる粒度で分割する

### 含めないもの
以下は今回 Common に入れない。
- Peer reveal ロジック
- AllowedIPs 生成
- WireGuard runtime 制御
- interface / route / firewall の実制御
- Group / User / Peer の WG ドメインモデル

### 期待成果
- 今後の別プロダクトで使い回せるベースができる
- wg-studio 本体の責務が整理される
- UI / Auth / Audit の見通しが良くなる

### 完了条件
- Common 化対象のディレクトリ構成が決まっている
- 既存画面が Common 参照へ置き換えられている
- 既存動作を壊していない
- 「これは Common」「これは wg-studio 固有」が明確になっている

## 3. 命名規則の統一

### 目的
現状の取っ散らかった命名を整理し、今後の保守性・可読性を上げる。

### 対象
- ディレクトリ名
- ファイル名
- コンポーネント名
- API 名
- state 名
- backend モジュール名
- settings キー
- UI 上のラベルと内部名の対応

### 方針
- 命名規則を先に定義してから直す
- 途中で命名ルールをぶらさない
- Common と wg-studio 固有で prefix / namespace 方針を分ける
- UI 表示名と内部識別子を必要に応じて分離する
- 「曖昧な名前」「責務が見えない名前」を減らす

### 例として整理したい観点
- `Common` という雑な名前で止めるか、より意味の強い名前にするか
- `settings`, `config`, `runtime`, `health` の役割境界
- router / service / component / store の命名揺れ
- dashboard / summary / overview の使い分け
- group route / allowed route / policy の用語統一

### 期待成果
- ファイル名や責務が初見で分かりやすくなる
- AI 補助時にも誤読が減る
- 今後の Common 化や plugin 化の障害が減る

### 完了条件
- 命名規則ドキュメントがある
- 主要ディレクトリ・主要コンポーネントが新ルールに沿っている
- 旧名が残っている箇所が把握できている

## v1.2.2 のスコープ外
今回やらないものを明示する。

- plugin 化の本実装
- nftables / routing / DNS / NAT の抽象化本体
- runtime health の大幅拡張
- RBAC の本格実装
- audit の高度化
- Next.js 移行本体
- 新機能追加を大きく広げること

---

2. Common 化
3. 命名規則統一

この順番を崩さないこと。

## 作業ゴール
v1.2.2 は「新機能を増やす版」ではなく、  
**v1.3 へ入る前の基盤整理版** と位置付ける。

期待する状態は以下。

- 直近で踏んだ不具合が塞がっている
- 再利用可能な共通部品が切り出されている
- 命名規則が整理され、今後の開発がしやすくなっている
- 次のバージョンで plugin 化や runtime health 強化へ進みやすい
