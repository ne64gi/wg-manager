# `1.x` Plan

Purpose: capture the current intended work order inside the `1.x` era.

Status: planning memo only.

## Core Rule

Within `1.x`, prefer structural separation before heavy feature layering.

Interpretation:

- separate internals before broadening behavior
- reduce future migration pain before polishing the surface
- leave authorization deeper in the sequence so it does not harden around the wrong structure

## Current Intended Order

### `1.1`

Theme: `Runtime separation`

Priority:

- highest urgency inside `1.x`

Rationale:

- Linux-specific runtime handling should be made replaceable early
- future Windows or other-runtime support depends on separation, not immediate portability
- delaying this raises migration cost for every later change
- the authorization entry seam should appear early, but full authorization behavior should not lead this phase

Key directions:

- adapter-style runtime boundaries
- platform-neutral core where possible
- launch-entry cleanup
- reduce direct coupling between core logic and current deployment assumptions
- isolate Linux-specific runtime control
- separate `config write`, `apply`, `status read`, and `artifact root` responsibilities
- introduce only the earliest authorization entry points that help later work:
  - no-op policy plugin
  - minimal action vocabulary
  - audit-ready authz decision recording
  - no broad deny matrix yet

### `1.2`

Theme: `Frontend modernization prep`

Priority:

- early, before the frontend becomes harder to move

Rationale:

- framework migration pain grows quickly when routing, data loading, layout, and browser-only behavior are tightly mixed
- the near-term goal is not immediate Next migration
- the real goal is a structure that can move to Next later without rewriting everything

Key directions:

- separate page composition from reusable UI
- reduce router coupling in shared components
- isolate browser-only APIs
- separate data fetching from presentation
- move toward Next-ready layout boundaries
- treat this as "Next-ready structure", not immediate framework migration

### `1.3`

Theme: `UX / visibility improvements`

Priority:

- after the runtime and frontend separation work has been established

Rationale:

- this is the point where operator value and usability should improve on top of cleaner internals
- current graph readability is weak and visibility tooling is still immature

Examples:

- click-to-expand graph views
- easier traffic inspection
- clearer visual hierarchy around charts
- apply-before-change diff visibility
- stronger desired / runtime drift visibility
- status and history improvements
- GUI navigation and workflow cleanup
- dangerous-operation confirmation improvements
- settings cleanup
- backup / restore

### `1.4`

Theme: `Authorization and audit hardening`

Priority:

- intentionally later than `1.1`, `1.2`, and the first round of operator UX work

Rationale:

- if authorization is hardened too early, it may lock in the wrong runtime or frontend boundaries
- first separate the system
- then add richer authz on top of cleaner seams

Scope examples:

- middleware + policy plugin hybrid
- action vocabulary cleanup
- broader route metadata coverage
- authz decision and audit integration
- fail-closed and default-deny hardening decisions
- real policy implementation
- audit coverage strengthening

## Cross-Cutting Quality Track

These should progress alongside `1.1` through `1.4`, not as their own version theme.

### Test expansion

- broaden Playwright beyond smoke where it helps each phase
- bundle ZIP verification
- JSON export/import round-trip coverage
- mobile workflow coverage
- auth expiration and apply-error failure-path coverage

### UI and wording audit

- translation-gap cleanup
- vocabulary drift cleanup
- internal value leakage cleanup
- docs and UI wording alignment
- stale-doc detection and cleanup

### External-facing docs and feedback readiness

- improve screenshot-backed README and landing docs before outreach
- add practical use-case docs such as home VPN, team VPN, and small office VPN
- prefer showing value through examples before wider community promotion

## Short Canonical Summary

Use this wording when discussing the current `1.x` order:

1. `1.1`: runtime separation
2. `1.2`: frontend modernization prep
3. `1.3`: UX and visibility improvements
4. `1.4`: authorization and audit hardening
