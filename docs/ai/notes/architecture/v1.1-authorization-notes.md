# v1.1 Authorization Notes

Purpose: capture the current direction for post-`v1.0.0` authorization work before implementation starts.

## v1.0.0 Acceptance Boundary

`v1.0.0` should be authorization-ready, not authorization-enabled.

Accepted in `v1.0.0`:

- middleware entry point exists
- route metadata can express action/resource intent
- policy plugin interface exists
- environment flags exist for future rollout
- default runtime behavior remains effectively unchanged unless operators explicitly enable the scaffold

Not required in `v1.0.0`:

- real RBAC policy
- enforced deny matrix
- production authorization plugin
- broad route coverage

## Current Decision Shape

Target direction:

- externalize authorization from the core control-plane logic
- keep `wg-studio` core focused on CRUD, config generation, apply, reveal, import/export, and status
- avoid baking a fixed RBAC model directly into route handlers or domain services

Planned design style:

- hybrid of hook-point authorization and middleware interception

Interpretation:

- middleware acts as the request-level orchestrator
- hook-point authorization handles action/resource-level decisions

This is intentionally not a pure route-only middleware design and not a pure inline permission-call design.

## Option A: External Policy Engine

Desired property:

- authorization logic lives outside the core code path

Core model:

- API handlers expose stable authorization hook points
- a loaded plugin decides whether a user may perform an action on a resource
- core code follows the result and stays policy-agnostic

Pseudo-flow:

```python
def check_permission(user, action, resource):
    for plugin in loaded_plugins:
        result = plugin.authorize(user, action, resource)
        if result is not None:
            return result
    return default_allow
```

Advantages:

- core code stays clean
- multiple authorization styles can coexist
  - role-based
  - attribute-based
  - LDAP-backed
  - future external identity adapters
- plugin failure does not require redesigning the domain model
- matches the goal of adding authorization after `v1.0.0`

## Option B: Middleware Plugin

Desired property:

- intercept requests before API code executes

Core model:

- FastAPI middleware inspects request context
- plugin decides allow or deny
- denied requests return `403`

Advantages:

- minimal route churn
- path-level policy can be centralized
- reusable for adjacent concerns
  - rate limit
  - IP allow/deny
  - operator maintenance gate

Limitations:

- resource-level decisions are harder if middleware is the only layer
- path-only matching is too coarse for many domain actions

## Preferred v1.1 Direction

Adopt a hybrid:

1. middleware layer for coarse admission
   - resolve authenticated login user
   - resolve route metadata into `action` and resource hints
   - call the policy engine
   - convert deny into `403`
2. hook-point checks for resource-sensitive operations
   - group read/update/delete
   - user read/update/delete
   - peer reveal/reissue/delete
   - config apply
   - state import/export

Reason:

- middleware gives a clean entry gate
- hook points give enough context for real authorization decisions
- avoids forcing resource lookup into middleware only

Middleware should stay thin:

- resolve subject
- resolve action/resource candidate
- invoke policy engine
- apply allow/deny result

Do not move business logic into the middleware itself.

## Non-Goals For First Authorization Iteration

- full enterprise IAM integration on day one
- redesigning the current domain model around roles
- introducing a hardcoded RBAC matrix into every route
- multi-tenant policy modeling
- policy UI editor in the first iteration

## Suggested Interface Shape

Keep the plugin contract narrow.

Possible contract:

```python
class AuthorizationPlugin:
    def authorize(
        self,
        *,
        user,
        action: str,
        resource_type: str,
        resource_id: str | int | None,
        resource_snapshot: dict | None,
        request_context: dict,
    ) -> bool | None:
        ...
```

Interpretation:

- `True` = explicitly allow
- `False` = explicitly deny
- `None` = plugin declines to decide, continue to next plugin or fallback

Recommended explicit decision model:

```python
class AuthzDecision(Enum):
    ALLOW = "allow"
    DENY = "deny"
    ABSTAIN = "abstain"
```

```python
class PolicyPlugin(Protocol):
    def authorize(
        self,
        subject: Subject,
        action: str,
        resource: Resource | None,
        context: dict,
    ) -> AuthzDecision:
        ...
```

## Suggested Action Vocabulary

Prefer stable action names over route-path strings.

Examples:

- `group.read`
- `group.create`
- `group.update`
- `group.delete`
- `user.read`
- `user.create`
- `user.update`
- `user.delete`
- `peer.read`
- `peer.create`
- `peer.update`
- `peer.delete`
- `peer.reveal`
- `peer.reissue`
- `config.apply`
- `state.export`
- `state.import`
- `audit.read`
- `settings.update`

## Route Metadata Direction

Avoid deriving everything from raw URL path matching only.

Prefer route metadata that maps each route to:

- action name
- resource type
- optional resource loader

Sketch:

```python
@router.delete("/peers/{peer_id}")
@authorize(action="peer.delete", resource="peer")
async def delete_peer(peer_id: int):
    ...
```

Middleware should consume this metadata and, when needed, call a resource loader to produce a richer resource snapshot for the policy engine.

This keeps endpoint bodies clean while still enabling resource-aware authorization.

## Suggested Rollout Order

1. define the smallest action vocabulary that supports future growth
2. add route metadata for sensitive routes
3. add middleware shell that orchestrates authz calls
4. add a no-op or reference plugin implementation
5. make authz decisions audit-visible even before broad enforcement
6. add hook points on sensitive mutating or secret-bearing operations
7. keep early enforcement minimal, such as admin-only allow on protected routes with clear bypasses
8. only then consider richer policy backends and broader deny/allow matrices

Practical interpretation:

- `1.1` should establish the decision point, plugin seam, and audit visibility
- `1.1` should not try to finish the real authorization model
- deeper deny/allow behavior belongs later, after runtime and frontend separation are less fragile

## Default Policy

Preferred direction:

- default deny for authenticated control-plane routes
- explicit allow for:
  - `GET /health`
  - setup flow
  - login flow
  - other intentionally public routes

Rationale:

- safer starting posture
- avoids accidental exposure from missing policy rules
- matches the long-term authorization goal better than default allow

## Failure Semantics

Preferred direction:

- fail closed when authorization is enabled and plugin execution fails
- keep a clearly separate mode if operators intentionally want authz-disabled behavior

Bypass set should stay minimal:

- `health`
- login
- setup

Do not silently allow privileged control-plane routes when the authorization layer is broken.

## Audit Expectations

Authorization should integrate with audit logging.

Target log shape:

- who attempted the action
- action name
- resource type and identifier
- allow / deny / abstain result
- plugin name or decision source

This is a natural fit for `wg-studio` because auditability is already part of the control-plane design.

## Why This Note Exists

This is a planning note, not an implementation commitment.

Its purpose is to preserve the intended design direction for `v1.1`:

- keep the core small
- keep authorization pluggable
- combine middleware and hook points instead of overcommitting to only one
- keep decision flow explicit: route metadata -> middleware orchestration -> policy engine -> audit log
