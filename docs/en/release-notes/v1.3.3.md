# Release Notes: `v1.3.3`

`wg-studio v1.3.3` finishes the UX and visibility work that started in `v1.3.0` and turns it into something much more practical for day-to-day operations.

## Highlights

- `Apply` now opens a generated server-config diff preview before the runtime update is confirmed
- added CLI / host-script support for `DB full backup / restore` and `state export / import`, clearly separating recovery from logical transfer
- added `pytest-safe.sh` plus a non-test database guard so tests and reset flows refuse to run against production databases
- upgraded `/network` into an operational relationship graph with selection focus, traffic hotspots, `Rx / Tx`, endpoint details, and persisted filters
- allowed selected `Group / User / Peer` nodes to be enabled or disabled directly from the network page
- clarified graph state rules so `selected / unselected / inactive / hidden / online` are visually distinct
- aligned search copy with the real search targets and added sortable headers to the Group / User / Peer lists
- added confirmations and warnings around dangerous actions such as `Peer reissue` and state import
- polished the shared shell, panels, cards, tabs, and navigation so the app reads more like a general-purpose admin console

## Why It Matters

This release is not about adding decoration. It is about making `wg-studio` safer to operate and easier to trust.

In practice that means:

- seeing what will change before applying it
- making destructive workflows harder to trigger by accident
- using the network graph as an operational surface instead of a static visualization

## Operator Notes

- destructive database work, import / restore, and manual SQL should always start with `scripts/backup-db.sh`
- `state export / import` is for logical transfer and should not be treated as a recovery-grade backup replacement
- `./scripts/stack.sh pytest ...` now routes through dedicated `*_test` / `*_e2e` databases
- `/network` filter state is persisted in localStorage and restored on the next visit
